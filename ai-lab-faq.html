AI Lab — Frequently Asked Questions

A technical deep-dive into AskSantorini.ai — the architecture, the engineering decisions and how an edge-native AI product is built and operated solo.

What is AskSantorini.ai?

AskSantorini.ai is an AI-powered travel concierge for the island of Santorini. It answers visitors' questions about where to eat, what to see and how to get around using live, source-grounded search, and turns that intent into real reservations through an affiliate discount-code booking engine. No app, no sign-up — visitors just ask.

Why is it built edge-native with no traditional server?

The entire platform runs on Cloudflare Workers — there is no server to patch, no container to maintain and no always-on database. Compute executes at the global edge close to each user, the system scales to zero when idle, and a push to the main branch ships instantly worldwide. The result is low latency, a minimal operational surface and predictable cost.

How does it give accurate, up-to-date answers instead of hallucinating?

Answers are generated by Google Gemini 2.5 Flash with Google Search Grounding, so responses are built from live, citation-backed information rather than frozen training data. Gemini Function Calling triggers structured, on-demand data fetches based on query intent, and the Google Places API supplies venue details, geocoding and directions.

How does the booking and affiliate system work?

Affiliates are managed in a Google Sheet — a source of truth a non-technical partner can edit directly — which syncs to Cloudflare KV for fast edge reads. When a visitor redeems a discount, the backend mints a reservation in Cloudflare D1, sends a mobile-optimized confirmation email through a Google Apps Script pipeline and notifies the partner. Booking codes are crypto-random with collision-retry.

How is the AI endpoint protected from abuse and runaway costs?

The chat endpoint fronts paid LLM and Places calls, so it is guarded by Cloudflare's native rate-limiting binding — an atomic primitive with no write bottleneck. An earlier KV-counter approach was rejected at the verification stage because eventually-consistent writes failed under burst. A secret-gated kill switch can instantly disable the assistant during any cost or abuse runaway.

How are changes deployed to production safely?

Production changes follow a strict branch, preview, verify and merge flow. The rollback target is identified before any live action, and each irreversible step has an explicit confirmation gate. This discipline has repeatedly caught bad changes at verification rather than in front of users.

Can the platform be reused for other destinations?

Yes. The architecture is being generalized into a shared worker core plus a per-instance config bundle — environment variables, persona, knowledge base, affiliate data and visual theme tokens — so a new destination launches as a config-and-theme file rather than a code fork. The first sibling instance ships with its own distinct AI persona, not a renamed clone.
