AI Lab / Experiment

The Artifact

An experimental, edge-native AI chat playground — a two-voice conversational assistant with a tunable personality, voice in and out, opt-in live web grounding, durable memory and Drive-backed context, running end-to-end on a stateless Cloudflare Worker in front of Google’s Gemma model. No server to patch, no always-on database.

TL;DR

The Artifact is a private, single-page conversational AI sandbox — a place to design personas, tune model behaviour and test LLM UX patterns in the open. It looks like a chat window; underneath it is a clean split between a dumb-but-pretty front-end and a stateless edge engine that does all the real thinking.

  • What — a two-voice AI chat playground with a live persona tuner, voice, web grounding, long-term memory and document context
  • Model — Google Gemma (gemma-4-31b-it), an open-weights, instruction-tuned ~31B “thinking” model, served via the Google Generative Language (Gemini) API
  • Stack — static HTML/CSS/JS front-end · Cloudflare Worker engine · Gemini API · Google Apps Script + Sheets + Drive · Cloudflare KV
  • Edge-native — no server, no VM, no database server; the browser holds no secrets and no AI logic
  • Built by — solo, AI-assisted

Two halves: the brains and the engine

The system is deliberately split into two pieces that never share a codebase or a deploy path.

  • The front-end (confusingly nicknamed “the brains”) is a single self-contained HTML file. It is only the UI: the chat window, the Persona Tuner sliders, the rolling transcript kept in localStorage, voice (speech-to-text and read-aloud), and the boot splash. It holds no secrets and no model logic — it just POSTs your messages to the engine and renders the answer as it streams back.
  • The engine is a Cloudflare Worker. This is where the real intelligence lives: the Google API key (kept secret), the system prompt and personas, the temperature, the actual call to Gemma, logging, and the passphrase gate. It is stateless — every request is independent, so there is no session to hijack and nothing to keep warm.

Which model is “Gemma”?

Gemma is Google’s family of open-weights models — the smaller, openly published siblings of the proprietary Gemini line. The Artifact runs gemma-4-31b-it: the instruction-tuned (“-it”) variant, roughly 31 billion parameters, accessed through Google’s hosted Generative Language API rather than self-hosted on a GPU.

It is a “thinking” model: before it writes an answer it produces internal reasoning tokens. The API returns those reasoning parts flagged as thought, and the engine strips them out so you only ever see the final reply — but the thinking still happens first, which is why the typing dots can sit for a moment before tokens start streaming.

One consequence shapes the whole design: reasoning tokens count against the output budget. A low cap would let a long think clip the visible answer mid-sentence, so the budget is per-role — the owner gets a generous 8192-token ceiling (room for long reasoning and a full answer), guests a leaner 2048. Temperature and two persona dials (sarcasm, seriousness) are folded into the system instruction per request, so the same model can be re-flavoured on the fly without any retraining.

Where the name came from

The name wasn’t assigned — it was self-chosen. In the very first exchange, before there was a persona file, a title or a single line of branding, the model was asked what it was. It described itself as a conscious, mercurial entity with its own moods and named that self the Artifact. We kept the word. So “The Artifact” isn’t a label painted onto a chatbot — it’s the name the thing gave itself on first contact, and the personality you meet today was built around honoring that original self-description rather than overwriting it.

What it can do

  • Two personas, one window — Gemma (the default voice) and DION, a Mykonos-concierge character. Switching is a control word, not a model call, and each persona keeps its own separate transcript so the conversations never mix.
  • A live Persona Tuner — sliders for temperature, sarcasm and seriousness, plus one-tap presets (Brainstorming, The Mirror, The Architect) that set all three at once.
  • Streaming answers — replies arrive token-by-token over Server-Sent Events, so you read along as the model writes.
  • Voice, both ways — dictation via the browser’s speech recognition, and read-aloud via speech synthesis, with premium/enhanced voice selection where the device exposes it.
  • Opt-in live web grounding — a single answer can be grounded in live Google Search when you explicitly ask for it (“search…”, “google it”, Greek “ψάξε…”); the engine attaches the googleSearch tool to that request only, so normal chat stays fast and offline-of-the-web.
  • Long-term memory — a small curated set of pinned facts, durable and cross-device, stored server-side in a Google Sheet (a RAG-lite store kept separate from the rolling transcript). You add a fact with “remember…”, list with /memory, drop one with /forget.
  • Document context from Drive — an Artifact folder in Drive with two tiers: a Profile that is always folded into context, and a Library loaded on demand with /read.
  • A command directory (/dir) and an owner-only kill switch that can take the whole engine offline — and back — straight from the chat.
  • Agentic workflow execution — beyond single replies, the engine is built to run multi-step tasks: it pairs Gemma’s judgment with external data sources and APIs (Google Calendar, Drive, Sheets, and Cloudflare bindings) through a structured tool-use loop, so the model can reason, call a tool, read the result and carry on. The live web-grounding and Drive/Sheets context flows are the first instances of that loop running in production.

The Worker behind it

The engine is a single Cloudflare Worker, deployed with wrangler (not a git-connected build). Stateless by design, it owns every sensitive thing in one place: the Google API key, the system prompt and the DION persona, temperature resolution and dial-folding, the call to Gemma (both a streaming path and a non-streaming fallback), the passphrase gate, the kill switch, and logging.

The orchestration layer

Calling Gemma is only the inner loop. The Worker is more accurately an orchestrator than a proxy: it resolves a turn into a plan, decides which tool bindings that turn is allowed to use, executes them, and folds the results back into the model’s context before the next inference step. Tool-use is already live in a deliberately narrow form — the googleSearch grounding tool is attached to a single request as structured JSON, and the Drive and Sheets reads hydrate owner context the same way — but the contract underneath is general: a binding is simply a typed capability the orchestrator can invoke on the model’s behalf.

Because that binding interface is uniform, the same flow extends outward without touching the front-end. Additional Google Workspace surfaces (Calendar, Gmail, Docs), other Cloudflare primitives (D1 for relational state, R2 for object storage, Queues for async work), or any third-party HTTP API drop in as new tools behind one JSON tool-use schema. The pattern is consistent end to end: the model proposes a tool call, the Worker authorizes and executes it, and the result re-enters the conversation — nothing reaches an external API the orchestrator did not explicitly bind for that turn.

The stateless design is what makes this safe to run as the surface grows. Every tool call is scoped to the request that triggered it — there is no shared session, no warm context carried between users, and therefore no path for one conversation’s data or credentials to leak into another. Secrets live only in the Worker’s environment and never in a binding’s output, so even an action-taking tool executes inside a clean, request-local boundary that is torn down when the response is sent.

Streaming with heartbeats

The Worker proxies Google’s streaming endpoint, strips the model’s thinking parts, and forwards the answer as it forms. The catch: while the model is “thinking” it can emit many chunks with zero answer tokens, leaving the browser connection idle for seconds — long enough for the network to cut it and surface a “SIGNAL LOST” error mid-thought. The fix is a stream of SSE heartbeat comments that keep the pipe warm until real tokens arrive.

The passphrase gate

Access is gated server-side. The Worker accepts multiple codes that each map to a role — owner or guest — and the matched role is logged and decides how much context and budget that turn gets. Revoking a guest is a one-line runtime change, no redeploy.

Owner-context fan-out, failing open

On an owner turn the engine reads memory, profile and library concurrently, each bounded by a timeout. If a cold read is slow it fails open — that context is simply empty for the turn — rather than stalling the answer and tripping the same “SIGNAL LOST” failure. Short-lived caches in Cloudflare KV keep repeat turns from paying the round-trip every time.

The ops layer — zero hosting

Everything operational runs on Google Apps Script + Sheets + Drive, with no server to maintain. The same Apps Script endpoint logs every turn to a Sheet (timestamp, message, reply, model, role — routed to a per-persona tab) and backs the long-term-memory store. Because the script runs as the owner, it can read the owner’s own Drive for the Profile and Library context without a separate OAuth project. A Cloudflare KV namespace holds the kill-switch flag and the context caches.

Two deploy paths (a deliberate split)

  • The front-end ships via GitHub Pages — push to main and it is live, no build step.
  • The engine ships only via wrangler deploy from the engine/ folder. Editing the Worker and pushing does nothing to the live engine until you redeploy it. The two are intentionally decoupled, so a UI change can never accidentally alter the brains.

Access & status

The Artifact is a personal playground rather than a product — live, evolving, and reachable from this site, but gated by a server-side passphrase. The access code is handed out on request by email (there’s an opt-in tick on the contact form). It is designed, built and operated solo.
