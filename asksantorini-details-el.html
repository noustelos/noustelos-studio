AskSantorini.ai

An edge-native AI travel concierge for the island of Santorini — conversational, source-grounded recommendations backed by a real affiliate booking engine, running end-to-end on serverless infrastructure with no host to patch and no always-on database.

TL;DR

A serverless AI travel concierge for Santorini that answers visitors from live, source-grounded search and turns that intent into real affiliate bookings — a consumer chat frontend, a transactional booking backend, and a zero-host ops layer, all running at the edge.

  • What — conversational concierge + affiliate discount-code booking engine
  • Stack — Cloudflare Workers · D1 · KV · Gemini 2.5 Flash + Search Grounding · Google Places · Apps Script + Sheets

Overview

AskSantorini.ai is a conversational assistant that helps visitors plan their time on the island — where to eat, what to see, how to get around — and turns that intent into actual reservations through an affiliate discount-code system.

It lives at the intersection of two things that don’t usually share a codebase: a consumer-facing LLM chat experience, and a transactional booking backend with partner management, email, and analytics. The entire stack runs at the edge — no traditional server, no container, no persistent VM.

It was built less as a one-off site than as the first instance of a reusable platform. The destination-specific layer — local knowledge, persona, partner data, and visual theme — is deliberately kept separate from the shared engine, so the same core can serve another destination by swapping configuration rather than forking code. A sibling instance, AskSingapore.ai, already runs on that shared codebase in early preview.

Built and maintained solo as an AI-assisted product: I own the architecture and the decisions; an AI coding agent handles execution under a strategy-first, gated workflow. The interesting part of that process isn’t the code generation — it’s the discipline around it (reviewable diffs, rollback-first deploys, verify-before-merge).

Tech Stack

Compute & edge

  • Cloudflare Workers — split into a static-assets frontend worker and a backend API worker
  • Zero-config CI/CD: push to main → instant global edge deploy

AI & data enrichment

  • Google Gemini 2.5 Flash with Google Search Grounding — live, citation-backed answers instead of frozen training-data knowledge
  • Gemini Function Calling — structured, on-demand data fetches driven by query intent
  • Google Places API — venue data, geocoding, directions

Persistence

  • Cloudflare D1 (SQLite at the edge) — reservation + booking records
  • Cloudflare KV — low-latency affiliate configuration store

Ops layer (serverless, zero hosting)

  • Google Apps Script — transactional email + analytics pipeline
  • Google Sheets as the affiliate source-of-truth, synced to KV on edit via an Apps Script trigger

Architecture

The system is intentionally split into two Workers with a clean separation of concerns:

  • A frontend worker serving the static client, git-connected for continuous deployment.
  • A backend worker that owns every binding — KV, D1, secrets — and exposes the chat + booking API.
[User Request]
      │
      ├──> (Frontend Worker) ──> Serves Static Assets (HTML/CSS/JS)
      │
      └──> (Backend Worker)  ──> [Lock: Production CORS]
                │
                ├──> Gemini 2.5 Flash + Search Grounding
                ├──> Cloudflare KV (Affiliate Data - Fast Reads)
                ├──> Cloudflare D1 (Reservations - SQLite Ledger)
                └──> Google Apps Script (Email & Analytics Pipeline)

The chat path is a single POST handler. A user message hits the backend, which composes a system instruction (the concierge persona), calls Gemini with Search Grounding for a current and sourced answer, and — depending on the detected intent — either enriches the response with Google Places data or routes to a lighter directions / “Open in Maps” flow. Cross-origin access is locked to the production origin so the backend never behaves as an open relay.

The booking side is where it stops being a demo. Affiliates are managed in a spreadsheet — the single source of truth a non-technical partner can actually edit — which syncs to KV for fast edge reads. When a user redeems a discount, the backend mints a reservation in D1, sends a mobile-optimized confirmation email through the Apps Script layer, and notifies the partner. Codes are generated, validated, and can be reissued, all at the edge.

Key Features

  • Source-grounded conversational concierge — a warm, deliberately un-gimmicky local persona that answers from live grounded search rather than stale model knowledge.
  • Affiliate booking engine — discount-code reservation flow, partner-editable affiliate management via Sheets → KV, plus transactional confirmation and partner-notification emails.
  • Intent-aware location handling — routes between rich Places results and a lightweight directions path based on what the user is actually asking for.
  • Enumeration-resistant codes — crypto-random booking codes with collision-retry, fully backward-compatible with the legacy format.
  • Operational kill switch — a secret-gated mechanism to instantly disable the assistant during a cost or abuse runaway, with a paired re-enable path.

Engineering Highlights

A handful of problems that were more interesting than they first appeared:

Choosing the right rate-limiting primitive

The chat endpoint fronts a paid LLM and paid Places calls, so an unthrottled abuse loop is a direct, metered cost leak. The first cut used the KV store as a per-IP counter — and it failed under exactly the burst it was built to stop. KV tops out around one write per second per key, meaning a flood against the same key creates an eventual consistency bottleneck where counter-writes get rejected; the fail-open safety path then lets everything through. The fix was the platform’s native rate-limiting binding — an atomic, purpose-built primitive with no write bottleneck. The takeaway, don’t build a high-frequency rate limiter on an eventually-consistent KV store, was caught at the verification step and never reached production.

Hardening edge static serving

Zero-config static serving is wonderfully convenient right up until you notice it will cheerfully serve your entire repo root — source, config, migrations — to anyone who guesses a path. The fix was an explicit asset-ignore manifest plus a standing rule: “is this file publicly reachable?” is now a deploy-time invariant, not an afterthought.

Gated, reversible deploys

Production changes follow a strict branch → preview → verify → merge flow, with the rollback target identified before any live action and an explicit confirmation gate at each irreversible step. More than once, that discipline caught a bad change at verification instead of in front of users — including the rate-limiter rewrite above.

A back office with nothing to host

Email, analytics, and the partner-facing admin surface all run on Apps Script + Sheets. The result: no server to patch, scales to zero, and a non-engineer can administer affiliates directly.

Roadmap

  • Multi-tenant edge platform. Finishing the clean split between the shared worker core and a per-instance config bundle — environment variables, persona / system-instruction, knowledge base, affiliate data, and visual theme tokens — so a new destination launches as a config-and-theme file rather than a fork. Each instance keeps its own distinct persona, not a renamed clone.
  • Tiered Places activation. Gemini Function Calling to fetch venue visuals on demand — text-only for broad queries, rich detail cards for specific venues — cutting both UI clutter and API spend.
  • Privacy & consent hardening. Enforced consent gating and data-deletion flows to a GDPR-grade standard.

Status

Live in production and iterating. Designed, built, and operated solo.
